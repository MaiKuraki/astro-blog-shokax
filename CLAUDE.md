# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Runtime and tooling

- Use **Bun** as the runtime/package manager (`packageManager: bun@1.3.6` in `package.json`).
- Node.js compatibility is not guaranteed by this project.

## Common commands

### Install

```bash
bun install
```

### Local development

```bash
bun run dev
```

### Build

```bash
# Full production build (Astro build + Pagefind index)
bun run build

# Site build only (used by Lighthouse CI)
bun run build:site
```

### Preview

```bash
bun run preview
```

### Type/content checks

```bash
bun run check
```

### Lint and format

```bash
bun run lint
bun run format
```

### Tests

```bash
# Run all tests
bun run test

# Run a single test file
bun test src/toolkit/posts/generateTagCloud.test.ts

# Run matching tests by name
bun test src/toolkit/posts/generateTagCloud.test.ts -t "pattern"
```

### Performance (same as CI)

```bash
bunx --bun lhci autorun --config=.lighthouserc.json
```

### Hyacine CLI entry

```bash
bun run hyc
```

## High-level architecture

### 1) Config-driven theme system (single source of truth)

- Main theme config lives in `src/theme.config.ts`.
- The config is strongly typed and normalized by `src/toolkit/themeConfig.ts` (`defineConfig(...)`), including theme color sanitization.
- Navigation, sidebar, layout mode, widgets, comments, AI features, and many UI toggles are controlled from this config.

### 2) Content model and data sources

- Astro Content Collections are defined in `src/content.config.ts`:
  - `posts`: `src/posts/**/*.{md,mdx}`
  - `moments`: `src/moments/**/*.{md,mdx}`
- `posts` schema includes blog metadata plus encryption-related fields (`encrypted`, `password`).

### 3) Routing and page generation

- Post pages are generated by `src/pages/posts/[...slug].astro`.
  - URL shape is `/posts/<post.id>/`.
  - It computes prev/next post navigation and related posts.
  - It handles encrypted posts at build-time and marks encrypted content to be excluded from Pagefind.
- Pagination is intentionally split:
  - Page 1: `src/pages/index.astro` -> `/`
  - Page 2..N: `src/pages/page/[page].astro` -> `/page/<n>/`
- `astro.config.mjs` enforces `trailingSlash: "always"`; keep internal links with trailing `/`.

### 4) App shell and layout composition

- Global shell is `src/layouts/Layout.astro`.
- `Layout.astro` mounts core interactive Svelte components (`NavBar`, `SearchPage`, `FloatingToolbar`) and chooses two-column vs three-column layout at runtime (`TwoColumnLayout.astro` / `ThreeColumnLayout.astro`) from theme config.

### 5) Markdown/MDX rendering pipeline

- Markdown/MDX behavior is centralized in `astro.config.mjs` (remark + rehype plugins, Shiki transformers, auto-imported MDX components).
- Runtime initialization for MDX interactive components (quiz/tabs) is in `src/toolkit/initMdxComponents.ts` and loaded from `Layout.astro`.

### 6) Search and indexing

- Search UI is `src/components/search/SearchPage.svelte` (Pagefind UI).
- Full indexing is generated only when running the full build command (`bun run build` includes `build:index`).
- Preserve Pagefind markers in content pages (`data-pagefind-body`, encrypted-page exclusions), especially in post routes.

### 7) Code block custom element flow

- `src/components/CodeBlock.svelte` is a custom element (`<code-block>`).
- Server-rendered code blocks are wrapped by `src/toolkit/wrapRenderedCodeBlocks.ts` so they become `<code-block><pre .../></code-block>`.
- `Layout.astro` renders a hidden `CodeBlock` instance so the custom element is defined on the client.

### 8) Styling and icon pipeline

- UnoCSS config is in `uno.config.ts`.
- It derives icon safelist entries from `src/theme.config.ts` so icon classes configured in theme/nav/sidebar are retained in builds.
- Keep using a relative import in `uno.config.ts` for theme config (do not switch to `@` alias there).

### 9) Hyacine integration

- Hyacine plugin wiring is in `hyacine.plugin.ts` (inject points + enabled plugins).
- Hyacine theme/content paths are declared in `hyacine.yml`.

## Repository-specific collaboration rules

These are from `.github/copilot-instructions.md` and should be preserved when assisting:

- Default to **Chinese** for agent output and code comments unless the user asks otherwise.
- Do **not** create extra Markdown report files to summarize work; summarize in chat.
- If adding dependencies, explicitly report package name, purpose, and usage scope.
- After code changes, run `bun run format` and `bun run lint`, then check IDE diagnostics and fix newly introduced issues.
